# 反风控改造思路

> 目标：让 Gigya / Treasure Data 风控系统认为脚本操作是真人行为，使登录评分稳定通过阈值。

---

## 零、项目里的监控工具说明

| 文件 | 数据来源 | 界面 | 什么时候用 |
|---|---|---|---|
| **`desk_monitor.py`** | 读本地 `data/accounts.csv` + `logs/run_*.log` | tkinter 半透明悬浮窗 | **批量运行期间**，看总进度：成功/失败数、当前跑到哪个账号、最新日志 |
| **`risk_overlay.py`** | **双重数据源**：① CDP WebSocket 直接订阅浏览器 Network 事件（Gigya/TD/SF/FB/GTM 全域名实时覆盖）；② tail `logs/risk_log.jsonl`（由 main.py Playwright 写入，含 Gigya 登录 errorCode / riskScore / botSuspected 精确解析）| tkinter 半透明悬浮窗 | **调试/采集期间**，看 Gigya 风控评分、errorCode、Cookie 状态、TD/FB/GTM 等回包，同时写入 `logs/risk_log.jsonl` 基线日志 |

**典型组合用法：**
```
python desk_monitor.py     # 看批量进度
python risk_overlay.py     # 同时采集风控基线数据（调试时才需要）
```

## 一、风控系统检测什么

根据已确认的接入方：**Gigya（SAP CDC）** + **Treasure Data**

| 检测维度 | 具体指标 | 当前脚本状态 |
|---|---|---|
| 鼠标行为 | 是否有移动轨迹、轨迹是否为贝塞尔曲线 | ✅ 三次贝塞尔曲线 + ease-in-out 速度 + 高斯抖动，全部走 CDP `Input.dispatchMouseEvent`（`isTrusted=true`） |
| 键盘行为 | 按键间隔分布、是否有完整 keydown/keyup 事件 | ✅ 正态分布节奏（μ=110ms σ=40ms）+ 4% 概率误输入后退格 |
| 页面停留 | 加载完到第一次操作的时间间隔 | ✅ `random_mouse_wander` + `page_idle_behavior` 随机停留 2~4s，等 Gigya 采集行为基线 |
| 导航路径 | 是否从主页自然导航到登录页 | ✅ 主页 → 等 Gigya SDK 就绪 → 点击「ログイン ／ 会員登録」→ 登录页 → 登录后自动导航（抽選履歴→介绍页→申込み一覧） |
| 滚动行为 | 页面加载后是否有滚动 | ✅ `random_scroll`（1~3 次随机滚动）及导航过程中多次 `scrollIntoView` |
| 元素聚焦 | 点击输入框前是否有 focus/hover 过渡 | ⚠️ `human_click` 含贝塞尔接近过程，但无显式 hover 停顿（P2.8，待优化） |
| 设备指纹 | Canvas/WebGL/AudioContext 每个账号是否一致 | ✅ Nstbrowser 已处理 |
| webdriver 标志 | navigator.webdriver | ✅ 已确认为 undefined |
| Cookie 历史 | 该设备/Profile 是否有正常浏览历史 | ⚠️ 每次清空 cookie，历史为零 |
| 请求频率 | 账号间操作间隔 | ❌ 无间隔，100个账号连续跑 |

## 二、改造优先级

### 🔴 P0（最高优先级，影响最大）

#### 1. 鼠标轨迹 ✅ 已完成
**方案**：三次贝塞尔曲线 + ease-in-out 速度曲线 + 中段高斯抖动，两段式点击（贝塞尔接近 + 精准修正），全部走 CDP `Input.dispatchMouseEvent`（`isTrusted=true`）。  
**代码位置**：`utils/anti_bot.py` → `_bezier_mouse_move()` / `human_click()`

#### 2. 页面加载后的空闲行为 ✅ 已完成
**方案**：主页、登录页均调用 `random_mouse_wander(moves=3~5) + page_idle_behavior()`，随机移动鼠标 + 偶尔滚动，持续 2~4 秒，让 Gigya 采集足够的行为基线再操作。  
**代码位置**：`modules/login_logic.py` → `login()` Step 3b / Step 4

#### 3. 导航路径还原 ✅ 已完成
**方案**：完整还原真人操作路径——  
```
goto(主页) → 等 Gigya SDK 就绪 → 点击「ログイン ／ 会員登録」
  → 登录页填表 → 提交 → MFA → 利用規約
  → マイページ
    → 点击「抽選履歴」标签
    → 点击「抽選履歴一覧を見る」
    → 点击「開催中の抽選一覧」图片
    → 向下滚动，点击「抽選へ進む」(a.goLotteryBtn)
    → lottery/apply.html（抽選応募一覧）
```
**代码位置**：`modules/login_logic.py` → `login()` Step 3b；`navigate_to_lottery()` Steps 9b-0~9b-5

---

### 🟡 P1（中优先级）

#### 4. 键盘输入节奏优化 ✅ 已完成
**方案**：正态分布节奏（μ=110ms σ=40ms），每隔若干字符随机停顿，4% 概率误输入后退格重输（含密码字段也做）。  
**代码位置**：`utils/anti_bot.py` → `human_type()`

#### 5. 账号间操作间隔
**问题**：100 个账号连续无间隔跑，短时间大量登录请求特征明显。  
**方案**：每个账号完成后随机等待 8~20 秒再开始下一个。  
**代码位置**：`modules/session_runner.py` → 账号循环内

#### 6. Gigya 初始化等待 ✅ 已完成
**方案**：主页和登录页均调用 `wait_for_function("() => window.gigya && window.gigya.isReady === true")`，超时静默继续（不阻断流程）。  
**代码位置**：`modules/login_logic.py` → `login()` Step 3b / Step 4

---

### 🟢 P2（低优先级，进一步优化）

#### 7. Cookie 历史保留策略
**问题**：每个账号都清空 Cookie，导致 Gigya 每次看到的都是全新设备，没有历史信誉。  
**方案**：只清除账号相关 Cookie（`gig_hasSession_*`、`sid` 等），保留设备识别 Cookie（`dwanonymous_*`）。  
**代码位置**：`modules/session_runner.py` → `clear_cookies()` 逻辑

```
当前：clear_cookies()（全清）
目标：只删除会话 Cookie，保留设备 Cookie
```

#### 8. 元素 hover 过渡
**问题**：点击前没有 hover 状态，真人通常鼠标先划过按钮再点击。  
**方案**：鼠标移动到元素后先停顿 200~500ms，触发 hover 状态，再点击。  
**代码位置**：`utils/anti_bot.py` → `human_click()`

#### 9. 滚动行为
**问题**：登录页从不滚动，真人通常会浏览页面。  
**方案**：页面加载后随机小幅滚动 1~3 次。  
**代码位置**：`utils/anti_bot.py` → `random_mouse_wander()`

## 三、效果验证方法

**不能打开 DevTools**（DevTools 本身是风控信号）。

验证方式：  
- **唯一指标：登录成功率**  
- 每次改动只改一处，跑 10 个账号，记录成功/失败数  
- 前后对比成功率，判断该改动是否有效  

建议改动顺序：
```
1. 先跑基线（当前代码）：记录 10 个账号成功率
2. 加鼠标轨迹 → 记录成功率
3. 加页面空闲行为 → 记录成功率
4. 改导航路径 → 记录成功率
5. 以此类推
```

## 六、下一步：先采集基线，再改造

### 为什么要先采集基线

在动任何代码之前，必须知道「现在的分数是多少」——否则改了之后也不知道有没有效果。  
`risk_overlay.py` 已经能实时看到数据，接下来要让它**把数据写进文件**，积累一个可对比的状态库。

### 要记录哪些字段

每次登录结束时，Gigya 会返回一个完整的 JSON。我们关心的字段：

| 字段 | 含义 | 判断依据 |
|---|---|---|
| `errorCode` | 登录结果 | `0` = 通过，其他 = 失败类型 |
| `riskAssessment.score` | 风控评分（0~100，越高越可信） | 低于阈值 → 被拦截 |
| `riskAssessment.allow` | 是否放行 | `false` = 直接拦截 |
| `botSuspected` | 是否被判定为 bot | `true` = 高危 |
| `UID` | 账号唯一 ID | 区分哪个账号的数据 |
| `timestamp` | 发生时间 | 排序、计算间隔 |

### 状态库格式：`logs/risk_log.jsonl`

每行一条 JSON，每次登录写一行：

```json
{"ts": "2026-03-05T14:22:01", "account": "xxx@gmail.com", "errorCode": 0, "score": 87, "allow": true, "botSuspected": false, "change": "baseline"}
{"ts": "2026-03-05T14:23:15", "account": "yyy@gmail.com", "errorCode": 403102, "score": 21, "allow": false, "botSuspected": true, "change": "baseline"}
```

`change` 字段手动标注当前是哪个版本（baseline / +mouse_move / +idle_wait 等），方便后续按版本分组对比。

### 采集流程

```
第一轮（baseline）
  └─ 不改任何代码，跑 10 个账号，记录 10 条 JSONL

第二轮（+P0.1 鼠标轨迹）
  └─ 只加鼠标贝塞尔移动，跑同批账号（或新批次），记录 10 条
  └─ 对比 score 分布：中位数有没有提升？botSuspected 比例下降了吗？

第三轮（+P0.2 页面空闲行为）
  └─ 在第二轮基础上叠加，再跑，再对比
  ...
```

### 立即要做的事

1. **✅ 已完成** `risk_overlay.py` 加了 `--log`/`--change` 参数，每条 Gigya 登录响应自动 append 到 JSONL
2. **跑基线**：10 个账号，不改任何反风控代码，收集原始分布
3. **看分布**：`score < 50` 的比例是多少？`botSuspected=true` 的比例？
4. 根据数据决定先做哪个 P0 改造

## 七、VPS 环境搭建步骤

> 前提：VPS 是 Windows 系统，Nstbrowser 已在上面运行，代理已配好。

### 第一步：安装 Python

1. 下载 Python 3.11（推荐，与本地开发版本一致）  
   → https://www.python.org/downloads/release/python-3119/  
   下载 `Windows installer (64-bit)`
2. 安装时勾选 **"Add Python to PATH"**，其余默认
3. 验证：
   ```
   python --version
   ```

### 第二步：克隆项目

```bash
git clone https://github.com/chenkeweii/autopokenmon.git
cd autopokenmon
```

### 第三步：建虚拟环境 + 装依赖

```bash
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
playwright install chromium
```

### 第四步：修改 config.py

以下几项必须改成 VPS 实际值：

| 参数 | 说明 |
|---|---|
| `NST_HOST` | Nstbrowser 监听地址，通常 `localhost:8848` |
| `NST_API_KEY` | Nstbrowser 里查 API Key |
| `PROXY_HOST` / `PROXY_PORT_*` | 实际代理服务器地址和端口范围 |
| `OTP_EMAIL_ADDR` / `OTP_EMAIL_AUTH_CODE` | 收验证码的邮箱和授权码 |
| `DO_CLICK_LOGIN` | 改为 `True`（生产模式） |
| `REQUIRE_OTP` | 按实际需求（通常 `True`） |
| `SIMULATE_IP_BAN_FROM_ACCOUNT` | 改为 `0` |
| `_TESTPAGES_BASE` | 改成 VPS 上的实际路径，或不用本地测试页则无所谓 |

### 第五步：跑基线数据采集

开两个命令行窗口：

**窗口 1（风控监控 + 写日志）：**
```bash
venv\Scripts\activate
python risk_overlay.py --log logs/risk_log.jsonl --change baseline
```

**窗口 2（主脚本，只跑 10 个账号）：**
```bash
venv\Scripts\activate
python main.py
```

跑完后 `logs/risk_log.jsonl` 里会有每次登录的风控数据。

### 第六步：查看基线结果

```bash
# 简单查看：直接打印所有记录
type logs\risk_log.jsonl

# 或用 Python 统计
python -c "
import json
lines = open('logs/risk_log.jsonl').readlines()
data = [json.loads(l) for l in lines]
scores = [d['riskScore'] for d in data if d.get('riskScore') is not None]
bots = sum(1 for d in data if d.get('botSuspected'))
print(f'样本数: {len(data)}')
print(f'riskScore 平均: {sum(scores)/len(scores):.1f}' if scores else 'score 无数据')
print(f'botSuspected=true: {bots}/{len(data)}')
print(f'errorCode 分布: { {d[\"errorCode\"]:data.count(d) for d in data} }')
"
```
